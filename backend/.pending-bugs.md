# Pending bug fixes — P2 / P3 follow-up list

Generated by Phase 8 of the multi-phase harness verification. P0 and P1
items have been applied; the items below are recorded for the next fix
pass. Each entry has a one-line description, a file:line reference, and
a short rationale.

The list mirrors items 10–25 from the Phase-8 brief. It is
deliberately markdown-only and **must not** import or reference any
runtime symbol — it is a static note for human reviewers.

---

## P2 — Important but not on the critical path

### 10. `harness/plugins/__init__.py` uses the legacy HookRegistry
- **File:** `backend/harness/plugins/__init__.py:165`
- **Issue:** Plugin hook registrations are routed through
  `harness.hooks.registry.hook_registry` (the deprecated object).
  The runtime dispatcher is `harness._hook_system.HookRegistry`, so
  plugin-registered handlers never fire in the modern pipeline.
- **Fix sketch:** Switch `_hooks` registration in `PluginContext.apply`
  to `harness._hook_system.get_hook_registry().register(...)`. Keep
  the legacy import for one release with a `DeprecationWarning` so
  downstream plugins can migrate.

### 11. `harness/tools/code_search_tool.py` still re-imports `json` inside the loop
- **File:** `backend/harness/tools/code_search_tool.py:31, 40, 48`
- **Issue:** Three `import json` calls live inside the per-iteration
  branch instead of at module top level. Harmless but wasteful.
- **Fix sketch:** Hoist `import json` to the top of the file with the
  other imports.

### 12. `harness/tools/dependency_graph_tool.py` only analyses `import_map` (no symbols)
- **File:** `backend/harness/tools/dependency_graph_tool.py:80-100`
- **Issue:** The "incoming" branch falls back to a substring search
  inside `import_map` keys when `reverse_map` is empty; for projects
  with many `__init__.py` shims this returns false positives.
- **Fix sketch:** Build `reverse_map` from the actual import lines
  and use it as the primary signal. Substring search should be a
  debug-only fallback.

### 13. `harness/tools/knowledge_graph_tool.py` doesn't cache `_find_json_graph`
- **File:** `backend/harness/tools/knowledge_graph_tool.py:50-130`
- **Issue:** Every `kg_search` call re-walks the disk probe list. For
  the typical three-probe CWD case this is sub-millisecond, but the
  agent_workspace glob expands to N children on every call.
- **Fix sketch:** Memoize the first successful path on `_find_json_graph`
  and short-circuit subsequent calls. TTL ≈ 5 minutes.

### 14. `harness/curator.py` ignores its `db` argument
- **File:** `backend/harness/curator.py:80, 145, 170`
- **Issue:** `run_curator`, `should_run`, `mark_run_completed` and
  `maybe_run_curator` accept (or could accept) a `db` parameter but
  never use it; the curator's audit trail lives only in JSON files.
- **Fix sketch:** Write a `curator_runs` row on each completion
  (timestamp, archived_count, stale_count, dry_run) using the existing
  `harness.memory.database` pool. Keep the JSON sidecar as the
  primary source of truth for boot-time reads.

### 15. `harness/curator.py` `should_run()` "first run seeds last_run_at" is fragile
- **File:** `backend/harness/curator.py:148`
- **Issue:** On a fresh install, `should_run()` returns `False` (never
  runs) but `mark_run_completed()` is never called, so the system
  *literally never* runs the curator until something else writes
  `_curator_last_run`. This is intentional, but undocumented and
  confusing.
- **Fix sketch:** Add a one-line docstring noting the seeding contract
  and ensure the lifespan startup writes a sentinel `_curator_last_run`
  so the next 168h boundary is computed from a known origin.

### 16. `harness/pricing_cache.py` `_CACHE_TTL_HOURS` is the only constant that matches its docstring
- **File:** `backend/harness/pricing_cache.py:21`
- **Issue:** Even after the Phase-8 P1 #9 docstring fix, the constant
  itself is fine but the **unit** (hours) is hidden; callers who
  compare against a `datetime` need to know whether to multiply by
  3600.
- **Fix sketch:** Rename to `_CACHE_TTL_HOURS = 168` is already
  correct; add a `CACHE_TTL_SECONDS = _CACHE_TTL_HOURS * 3600` alias
  for callers and use it in the "is_stale" check.

### 17. `harness/_hook_system.py:discover_plugins` swallows filesystem errors silently
- **File:** `backend/harness/_hook_system.py:380-403`
- **Issue:** `except Exception` blocks log at `WARNING` and continue,
  which is the right policy for plugin setup but masks obvious config
  errors (e.g. a plugin's `plugin.yaml` with invalid YAML).
- **Fix sketch:** Distinguish manifest-parse errors (log at `ERROR`
  with a stack-trace hint) from setup-time errors (log at `WARNING`
  as today). The `format_scan_report` helper in `skills_guard.py` is
  a good model.

### 18. `harness/mcp/client.py` `MCPClient` constructor takes no config
- **File:** `backend/harness/mcp/client.py:687`
- **Issue:** All config (timeouts, max_rpm, etc.) lives on
  `SamplingHandler`. The outer `MCPClient` should expose a `config`
  attribute so `set_llm` can be wired through it.
- **Fix sketch:** Add `MCPClient.__init__(self, sampling_config: dict | None = None)`
  that propagates the dict to `SamplingHandler`.

### 19. `api/routers/curator_api.py:_read_curator_state` is missing
- **File:** `backend/api/routers/curator_api.py:25`
- **Issue:** The router imports `_read_curator_state` from
  `harness.curator`, but that symbol only existed as `_load_state`.
  Phase 8 added `_read_curator_state` as an alias to `_load_state`,
  so this is now resolved; record the original gap so the symbol
  policy is documented.
- **Fix sketch:** None — already addressed by Fix #2.

### 20. `api/main.py` lifespan swallows SandboxManager construction errors
- **File:** `backend/api/main.py:196-198`
- **Issue:** `SandboxManager()` is called without a try/except. A
  missing Docker daemon (very common in CI) will abort startup
  instead of degrading to "no-sandbox" mode.
- **Fix sketch:** Wrap the `SandboxManager()` call in
  `try/except Exception` and set `sandbox_manager = None` on
  failure, with a `logger.warning`.

## P3 — Nice-to-haves, polish, and longer-term improvements

### 21. `harness/prompt_builder.py` `_skills_cache_path` writes to home but reads from a const
- **File:** `backend/harness/prompt_builder.py:28, ~250`
- **Issue:** `_SKILLS_CACHE_FILE` is a module-level constant but
  `_skills_cache_path()` (a function) returns a slightly different
  path. Phase 8 confirmed the writer and reader both use the same
  file, but the indirection is confusing.
- **Fix sketch:** Replace `_skills_cache_path()` with a direct
  reference to `_SKILLS_CACHE_FILE`; mark the function deprecated.

### 22. `harness/tools/skill_tools.py:_adapt_skill` hard-codes replacements
- **File:** `backend/harness/tools/skill_tools.py:67-86`
- **Issue:** The Anthropic→TestAI tool-name mapping is a hand-rolled
  list. New tool additions require code changes.
- **Fix sketch:** Move the table to `data/tool_aliases.yaml` and load
  it on demand. Caching is fine — the table is small.

### 23. `harness/tools/skills_guard.py` `INSTALL_POLICY["community"]` is too strict
- **File:** `backend/harness/tools/skills_guard.py:45`
- **Issue:** Community-sourced skills are blocked on any
  "caution"-level finding, which makes Phase-8 Fix #1 effectively
  install-blocking for ~80% of community skills. Real-world skills
  often have legitimate `Read`/`Edit` patterns flagged as caution.
- **Fix sketch:** Split the "caution" verdict into
  "caution-network" (block) and "caution-shell" (warn but allow with
  user confirmation). Roll out behind a feature flag.

### 24. `harness/mcp/client.py` `SamplingHandler.metrics` is a plain dict
- **File:** `backend/harness/mcp/client.py:605`
- **Issue:** `metrics = {"requests": 0, ...}` is mutated from the
  sampling coroutine without a lock. Concurrent samples can drop
  increments.
- **Fix sketch:** Switch to `threading.Lock` (or
  `contextvars.ContextVar` for asyncio-friendly semantics).

### 25. `harness/_hook_system.py` `get_hook_registry()` mutates a global
- **File:** `backend/harness/_hook_system.py:151`
- **Issue:** The function reads/writes a module-global
  `_hook_registry`. In a multi-tenant deployment this leaks hook
  registrations across requests.
- **Fix sketch:** Store the registry on `app.state` (FastAPI) or pass
  it through the agent's `AgentDependencies`. Keep the module-global
  for back-compat but add a deprecation log on access.
