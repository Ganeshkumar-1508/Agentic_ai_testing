# C08 — JobSpec canonicalisation

**Date**: 2026-06-21
**Status**: Implemented (backend MVP)
**Decision tree**: `docs/2026-06-21-architecture-decision-tree.md#c08`

## Problem

The June 18 2026 audit found three separate frontend flows bypass
the `JobSpec` handoff that the domain glossary says is canonical:
Agent page → `/api/agent/run`, Pipeline → Orchestrate →
`/api/delegate`, Pipeline → Quick Test →
`/api/pipeline/from-requirements`. None of them construct a
`JobSpec`. The chat Role's `submit_job` tool is the only place
that does. Deletion test: if you delete the `JobSpec` concept,
none of the three flows breaks — proving the seam is
hypothetical, not real.

The 12 fields of `JobSpec` (at `harness/jobs/spec.py:108`) are
justified — TestAI is testing-vertical with fine-grained control
needs. But the from-requirements flow has 30+ test-specific
fields stuffed into a `dict[str, Any]` context, the `tier` field
isn't on every endpoint, and the chat's `submit_job` doesn't
share a code path with the API endpoints.

## Decision (locked, 10 questions)

| Q | Decision | Rationale |
|---|----------|-----------|
| Q1 | Canonicalize at orchestrator boundary, not API | The 3 API surfaces have different UX requirements; JobSpec is the handoff to the orchestrator. |
| Q2 | Optional `tier` field with default=1 (all endpoints) | Non-breaking. Tier=1 is the right default for all paths. |
| Q3 | from-requirements extras in `JobSpec.context.test_config` (typed sub-model) | The 30+ fields ARE test-specific; orchestrator ignores them. |
| Q4 | Pydantic `JobContext` with `extra='allow'` | Strict for known, permissive for unknown. |
| Q5 | All four durable, uniform path | ~5ms DB write for huge durability benefit. |
| Q6 | New `POST /api/jobs` accepts JobSpec directly | Per user constraint: "no legacy. remove them or rewrite them." |
| Q7 | Remove + rewrite in same sprint | (out of scope for this MVP — see open questions) |
| Q8 | 8 chat tools: submit_job + 7 read/control ops | Full context for the LLM. |
| Q9 | Extend `JobSpecStore` with 7 new methods | One store owns the lifecycle. |
| Q10 | `list_jobs` returns `JobSummary` (spec summary + latest run) | Actionable signal, no follow-up calls. |

## Implementation

### Modified: `backend/harness/jobs/spec.py`

Added C08's typed `JobContext`:

```python
class TestConfig(BaseModel):
    model_config = ConfigDict(extra="allow")
    pre_commands: list[str] = []
    post_commands: list[str] = []
    cache_directories: list[str] = []
    browser: str = ""
    os: str = ""
    runtime_version: str = ""
    timeout_seconds: int = 0
    parallel_jobs: int = 1
    retry_count: int = 0
    tags: list[str] = []

class JobContext(BaseModel):
    model_config = ConfigDict(extra="allow")
    session_id: str | None = None
    agent_id: str | None = None
    test_config: TestConfig | None = None
    request_metadata: dict[str, Any] | None = None
```

`JobSpec.context` is now a `JobContext` (or dict for legacy
callers). The `to_record()` helper serializes it via
`JobContext.to_payload()`. A plain-dataclass fallback is
provided for environments where pydantic isn't installed.

The `from_chat_submission` classmethod was updated to build
a `JobContext` automatically. `from_dict()` and `to_dict()`
handle the roundtrip (Pydantic → dict → Pydantic).

### Modified: `backend/harness/store/protocols.py`

Added 4 new dataclasses for the chat-facing 8-tool surface:
`JobSummary`, `JobStatus`, `JobComment`, `JobOutput`.

Extended `JobSpecStore` with 7 new methods (Q9):

- `list_by_session(session_id, *, limit=20) -> list[JobSummary]`
- `get_status(spec_id) -> JobStatus | None`
- `cancel(spec_id) -> bool`
- `pause(spec_id) -> bool`
- `resume(spec_id) -> bool`
- `add_comment(comment: JobComment) -> None`
- `get_output(spec_id) -> JobOutput | None`
- `list_comments(spec_id, *, limit=50) -> list[JobComment]` (companion, not in the 8-tool surface)

The original 4 methods (`save`, `get`, `update_status`,
`list_pending`) are unchanged.

### New file: `backend/harness/jobs/submitter.py` (~100 lines)

The shared `submit_job_to_orchestrator(spec, *, job_spec_store,
orchestrator_engine_factory)` function. Two responsibilities:

1. **Persist** the spec to `JobSpecStore` (durable across restarts).
2. **Dispatch** to `OrchestratorEngine.run_job_spec(spec)`.

Returns the `run_id` (may be `""` if dispatch failed but
persistence succeeded — caller can poll for recovery).

The function is the single entry point for all four submission
paths (chat, `/api/agent/run`, `/api/delegate`,
`/api/pipeline/from-requirements`). They all become thin
wrappers that build a `JobSpec` and call this function.

### New file: `backend/api/routers/jobs.py` (~280 lines)

The new `POST /api/jobs` surface, plus 7 chat-facing endpoints:

- `POST /api/jobs` — submit a JobSpec
- `GET /api/jobs/{spec_id}` — full spec detail + comments
- `GET /api/jobs?session_id=X` — list `JobSummary` for a session
- `POST /api/jobs/{spec_id}/cancel`
- `POST /api/jobs/{spec_id}/pause`
- `POST /api/jobs/{spec_id}/resume`
- `POST /api/jobs/{spec_id}/comments` — add a comment
- `GET /api/jobs/{spec_id}/output` — get the job's output

The router reads the `JobSpecStore` and `OrchestratorEngine`
factory from `app.state`, with fallback to the module-level
injector. Returns 503 if no store is wired (clean error for
mis-configured deployments).

### New file: `backend/tests/test_c08_jobs.py` (30 tests, all passing)

Covers:
- `JobContext` (Pydantic with `extra='allow'`) — 5
  (well-known fields, unknown fields, nested `TestConfig`,
  roundtrip)
- `JobSpecStore` extensions (Q9) — 9
  (`list_by_session` with truncation, `get_status`,
  `cancel` (true/false), `pause`/`resume` roundtrip,
  `add_comment`, `get_output`)
- Shared submitter (Q5) — 6
  (persists + dispatches, dispatch failure returns empty but
  persists, no store, no orchestrator, `new_spec_id`)
- FastAPI router (Q6, Q8) — 10
  (`POST /api/jobs` happy path, rejects empty prompt,
  `queued` status on dispatch failure, `GET /api/jobs/{id}`,
  404, list with session filter, list requires session_id,
  cancel/pause/resume, comments + output, 503 on no store)

## Behavior change (observable)

| | Before C08 | After C08 (this MVP) |
|---|---|---|
| 4 submission paths | chat via JobSpec; 3 others bypass it | chat via JobSpec; 3 others still bypass (TODO: rewrite as wrappers per Q7) |
| Context shape | `dict[str, Any]` (untyped) | Pydantic `JobContext` with `extra='allow'` |
| from-requirements 30+ fields | dumped into the same dict as `session_id` | separated into `context.test_config` (typed) |
| Chat LLM tools | 4 (submit, list, get, cancel) | 8 (+ pause, resume, comment, output) |
| `JobSpecStore` surface | 4 methods | 11 methods |
| New external API | none | `POST /api/jobs` (the canonical A2A endpoint) |

## Failure modes

| Failure | Detection | Behavior |
|---|---|---|
| Pydantic not installed | `ImportError` at module load | Falls back to plain-dataclass `JobContext`/`TestConfig` |
| `JobSpecStore` not wired | `app.state.job_spec_store is None` | Router returns 503 |
| `OrchestratorEngine` not available | factory returns `None` | Spec persisted, `run_id=""` returned; user can poll |
| Dispatch raises | caught in `submit_job_to_orchestrator` | Spec persisted, `run_id=""` returned, error logged |
| `pause` on already-terminal job | store returns `False` | Tool returns `paused=false` (caller decides what to do) |
| `cancel` on completed job | store returns `False` | Tool returns `cancelled=false` |
| Missing `prompt` in POST | FastAPI validation | 400 with detail |

## Open follow-ups (deferred per Q7)

Per C08 Q7 (locked): the 3 legacy endpoints and the frontend
rewrite should be done in the same sprint. **For this MVP, the
legacy endpoints still exist and the frontend still uses them.**
The new `POST /api/jobs` is the canonical path; the legacy
endpoints can be migrated one at a time:

1. **`/api/agent/run`** — current 5 fields → build `JobSpec`,
   call `submit_job_to_orchestrator`, return `{spec_id, run_id}`.
2. **`/api/delegate`** — current 11 fields → same pattern. The
   `tasks`, `toolsets`, `mcp_servers`, `role` fields become part
   of the `JobSpec.capabilities` (or move to a separate
   `JobSpec.context.delegation` field).
3. **`/api/pipeline/from-requirements`** — the 30+ fields map
   directly to `JobSpec.context.test_config` (already a typed
   `TestConfig`).
4. **Frontend rewrites** — `src/lib/api/jobSpec.ts` with three
   `toJobSpecFromX(state)` helpers; each page calls `POST /api/jobs`.
5. **Delete legacy endpoints** — once all three call sites use
   the new endpoint.

Other follow-ups from the decision tree (Q1-Q10):
- Pause/resume semantics in the orchestrator loop (currently
  the store marks `paused` but the orchestrator doesn't poll —
  this is a deeper change).
- `add_comment` schema: free-form text vs structured
  (`kind`, `author`, `body`)? We chose structured.
- Pagination on `list_jobs`: cursor-based? offset? We cap at 20
  for v1.
- `get_output` for in-progress jobs: returns empty `JobOutput`
  for now (status="pending"). A real implementation would
  return partial output.
- `cancel_job` propagation: does it cancel running subagents?
  Currently the store just marks the job cancelled; the
  orchestrator doesn't observe.
- C02 (Agent Teams) interaction: does the team lead also have
  these 8 tools? Per the open question in the C02 design
  doc — currently no.
