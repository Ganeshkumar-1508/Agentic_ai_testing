# C07 — ThrottlePolicy Wire (4-Step Budget Auto-Throttle Ladder)

**Date**: 2026-06-22
**Status**: implemented
**Owner**: TestAI Harness
**Reference**: `architecture-review-2026-06-21.html` §C07 (Sprint 1-2 foundation gap)

---

## Goal

Wire the 4-step budget auto-throttle ladder that's defined in `harness/budget_tracker.py` but never actually applied. Today the tracker computes the step; the side effects (HITL gate, sequential-only, cheaper model, pause) sit in the snapshot but nothing reads them. A run can blow through the hard cap and only get noticed when the LLM provider returns 429.

The ladder is the same pattern Hermes uses for `cron.3-minute hard interrupt`: graduated friction — each step buys more headroom at the cost of more user friction. The user always knows which step the system is on (via the activity feed + jobs page UI).

---

## The 4-step ladder

| Step | Side effect | Hook | Cost to user |
|------|------------|------|--------------|
| 0 | ok | — | none |
| 1 | HITL | `agent.set_hitl_gate(True)` → every tool call goes through `PermissionManager.await_approval` | 1 click per tool call |
| 2 | sequential | `agent.set_sequential_only(True)` | no cost today (chat is already sequential); flag is a no-op for chat, meaningful for sub-agents that fan out |
| 3 | cheaper model | `llm_router.set_tier("small")` → subsequent LLM calls use the leaf-class model | lower-quality completions |
| 4 | pause | `agent.interrupt()` → the run loop raises on next `_check_interrupt()` | full stop |

The default thresholds (in `BudgetConfig`):

| Threshold | Default | Notes |
|-----------|---------|-------|
| `hitl_threshold_usd` | $1.00 | soft warning — user gets approve/deny on each tool |
| `sequential_threshold_usd` | $2.00 | only meaningful for fan-out patterns |
| `cheaper_model_threshold_usd` | $3.00 | swap to a smaller model for the rest of the run |
| `pause_threshold_usd` | $4.00 | hard pause before the run_hard_cap at $5 |
| `run_soft_cap_usd` | $1.50 | (legacy, used in dashboards) |
| `run_hard_cap_usd` | $5.00 | abort threshold — orchestrator checks before dispatching |

## Why now (triggers)

- **Silent budget blowouts** — a misbehaving agent can rack up $30 in a few minutes before the user notices.
- **No user signal** — the throttle step is computed but never visible.
- **Sprint 1-2 gap** — the report flagged this as a foundation gap; C01-C06 are done, C07 is the last unstarted item on the Sprint 1-2 list.
- **Ladder is a one-liner away from being useful** — `LLMRouter.set_tier()` already exists; `Agent.interrupt()` already exists; we just need a `tracker.observe(agent, llm_router, event_bus)` to glue them.

## Design: `BudgetTracker.observe()` — the single glue

```python
async def observe(
    self,
    *,
    agent: "Agent | None" = None,
    llm_router: "LLMRouter | None" = None,
    event_bus: "EventBus | None" = None,
) -> "BudgetSnapshot":
    """Re-evaluate the throttle step and apply the side effects.
    
    Called by the Agent's run loop after every LLM call. Returns
    the post-evaluation snapshot. If the step climbs, emits a
    `budget.throttled` stream event with the new step + which
    side effects were applied.
    """
    prev = self._throttle_step
    new = self.check_soft_cap()
    if new == prev:
        return self.snapshot()
    
    # Apply side effects.
    if agent is not None and new >= 1:
        agent.set_hitl_gate(new >= 1)
    if agent is not None and new >= 2:
        agent.set_sequential_only(new >= 2)
    if llm_router is not None and new >= 3:
        llm_router.set_tier("small")
    if agent is not None and new >= 4:
        agent.interrupt()
    
    # Emit event.
    if event_bus is not None:
        await event_bus.emit(BudgetThrottleEvent(
            run_id=self.run_id,
            session_id=self.session_id,
            prev_step=prev,
            new_step=new,
            spent_usd=self._spent,
            soft_cap_usd=self.config.run_soft_cap_usd,
            hitl_active=new >= 1,
            sequential_active=new >= 2,
            cheaper_model_active=new >= 3,
            pause_requested=new >= 4,
        ))
    
    return self.snapshot()
```

The `Agent` doesn't need to know about the tracker; the orchestrator passes it explicitly. This keeps the tracker a **policy primitive** that any caller (orchestrator, delegate_task, custom test loop) can drive.

## Agent hooks

Three new methods on `Agent`:

```python
def set_hitl_gate(self, value: bool) -> None:
    self._hitl_gate = bool(value)
    if value:
        # Also flag the permission manager so even tools that
        # bypass the dispatcher go through approval.
        self._deps.permissions.set_force_approval(True)

def set_sequential_only(self, value: bool) -> None:
    self._sequential_only = bool(value)
    # No-op for the chat surface (tools already run one at a time);
    # the flag is consumed by delegate_task to demote fan-out.

@property
def hitl_gate_active(self) -> bool:
    return getattr(self, "_hitl_gate", False)

@property
def sequential_only_active(self) -> bool:
    return getattr(self, "_sequential_only", False)
```

The `interrupt()` method already exists for step 4.

## ToolDispatcher hook

`tool_dispatch.py::dispatch` reads `agent.hitl_gate_active` before deciding whether to call `await_approval`:

```python
# If the budget-tracker HITL gate is on, force approval
# regardless of the per-tool policy. This is how step 1
# buys the user the "approve every tool call" friction.
force_hitl = agent is not None and agent.hitl_gate_active
if force_hitl or self._policy.requires_approval(tool_name, args):
    approved = await self._permissions.await_approval(...)
```

This is **additive**: existing per-tool approval policies still work. The throttle gate just forces approval on top.

## Orchestrator wiring

The orchestrator's `run_single` builds the `BudgetTracker`, sets it on the contextvar, and **hands it to the Agent's run loop**. The Agent calls `tracker.observe(agent=self, llm_router=self.llm, event_bus=self._event_bus)` after every LLM call.

The orchestrator also calls `tracker.check_soft_cap()` once before dispatching, to abort early if the cap is already exceeded (the "session-level" pre-check). The per-LLM-call check inside the Agent is the fine-grained one.

```python
# In OrchestratorEngine.run_single (pre-coordinator):
tracker = BudgetTracker(run_id=run_id, session_id=session_id)
if tracker.hard_cap_exceeded():
    return {"status": "budget_exceeded", "error": "session budget already exhausted"}
budget_token = set_current_tracker(tracker)
# Hand to the Agent:
dt._budget_tracker = tracker
```

The Agent's run loop (the per-LLM-call path) calls:

```python
# After every LLM call (in the message-append step):
tracker = getattr(self, "_budget_tracker", None)
if tracker is not None:
    await tracker.add_cost(model=model, input_tokens=..., output_tokens=...)
    await tracker.observe(
        agent=self, llm_router=self.llm, event_bus=self._event_bus,
    )
```

This means: the LLM router doesn't need to know about the tracker; the Agent does, in exactly one place (the message-append step). Clean, testable, no circular dependency.

## Design decisions (grilled 2026-06-22)

1. **Tracker as a policy primitive, not a side-effect carrier** — `BudgetTracker.compute_step()` is pure; `observe()` is the only place that touches `Agent` / `LLMRouter` / `EventBus`. This makes it testable without spinning up an Agent.
2. **One call site for `observe()`** — the Agent's run loop, after every LLM call. Not the orchestrator, not the dispatcher, not the LLM router. Single call site = single source of truth.
3. **HITL gate is additive** — `agent.set_hitl_gate(True)` doesn't disable existing per-tool policies; it forces approval on top. The user gets the strongest approval posture the system can give.
4. **Step 2 is a no-op for the chat surface** — tools already run sequentially. The flag is exposed for sub-agents that fan out (a future deepening); for now it's a marker that the dashboard can show.
5. **`agent.interrupt()` already exists** — step 4 reuses the existing interrupt mechanism. No new Agent API surface for the pause step.
6. **Stream event is the activity-feed source of truth** — the UI subscribes to `budget.throttled` events rather than polling the snapshot. Real-time, no polling, the user sees the step climb as it happens.
7. **Throttle ladder is sticky** (de-escalation off) — once a run climbs to step 3, it stays at 3 for the rest of the run. De-escalation would be confusing for the operator (per the existing `BudgetTracker.check_soft_cap` docstring).
8. **UI design uses design-taste-frontend** — Off-Black, Geist, 1 accent color, no emojis, asymmetric pip ladder.
9. **Per-step event emit, not on every LLM call** — the event is only emitted when the step *climbs* (i.e. `new > prev`). Otherwise we'd flood the activity feed with redundant `step 0 → step 0` events.
10. **`tracker.observe()` is async** — the event emit is async (the EventBus API). The Agent's call site awaits it; a failure to emit never breaks the run.

## What this fixes

- **Silent budget blowouts** — the throttle fires *during* the run, not after.
- **No user signal** — the activity feed shows the step climb in real time.
- **No automatic friction** — today, a runaway agent runs until the LLM provider refuses; C07 puts HITL on the next tool call the moment spend crosses the threshold.

---

## What this does NOT fix (deferred)

- **Per-user-per-day / per-org caps** — the `GlobalBudgetTracker` in `harness/cost_tracker.py` handles these at the LLM-router level. C07 only wires the per-run ladder. Cross-run aggregation is a follow-up.
- **Custom threshold per agent type** — today every run uses the same `BudgetConfig` defaults. A future sprint could let the user set "this is a heavy batch run, raise my caps to $20".
- **Auto-resume after step 4** — the user can manually re-submit. Auto-resume would require a new "throttle_pass" event and a smarter orchestrator.
- **Step 2 enforcement in sub-agent fan-out** — the flag exists; delegate_task doesn't read it yet. Future deepening.

## Wire-level example

```http
# 1. Run starts. Tracker at $0, step 0.
# 2. Coordinator calls LLM 5 times. add_cost() called. check_soft_cap() → 0.
# 3. Coordinator calls LLM 10th time. add_cost() makes spent $1.04.
#    observe() → step 1 (HITL). agent.set_hitl_gate(True) called.
#    activity feed emits: {"type": "budget.throttled", "new_step": 1, ...}
# 4. Next tool call from the coordinator: ToolDispatcher sees HITL gate.
#    Forces await_approval(). User clicks "Approve" on the dashboard.
# 5. Coordinator continues. More LLM calls. add_cost() → $2.13.
#    observe() → step 2 (sequential). agent.set_sequential_only(True) called.
#    No-op for chat; flag stored.
# 6. Coordinator at $3.04. observe() → step 3 (cheaper model).
#    llm_router.set_tier("small") called. All subsequent LLM calls use the leaf model.
# 7. Coordinator at $4.02. observe() → step 4 (pause).
#    agent.interrupt() called. Next _check_interrupt() raises RuntimeError.
#    Run ends. Activity feed shows "Run paused at step 4 — $4.02 of $5.00".
```
