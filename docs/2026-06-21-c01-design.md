# C01 — Worktree isolation

**Date**: 2026-06-21
**Status**: Implemented (minimum viable)
**Decision tree**: `docs/2026-06-21-architecture-decision-tree.md#c01`
**Reference**: `reference/OpenHarness/src/openharness/swarm/worktree.py:135`

## Problem

Two parallel subagents in TestAI today clobber each other's files
because they share the same `/workspace/repo` directory. The kanban
board serializes through WIP limits, but this is a pass-through cost
— the underlying filesystem is shared. The June 18 2026 audit
flagged this as a `High` gap.

The orchestrator itself can clobber the user's working tree if it
runs write tools (e.g. `commit_and_open_pr`). With multiple active
orchestrator runs, two PR-creating runs would race on the same
branch.

Claude Code, OpenHarness, and ECC2 all solve this with git
worktrees. The 2026 production pattern is **PR-based, not
fast-forward** — each worktree-isolated work unit pushes its branch
and opens a draft PR. Tier-1 auto-merges; tier-2/3 awaits human
review.

## Decision (locked, 6 questions)

| Q | Decision | Rationale |
|---|----------|-----------|
| Q1 | Hybrid (per-session + per-subagent) | Claude Code v2.1.x production pattern. Per-session protects the user's tree; per-subagent protects siblings. |
| Q2 | In container's volume at `<repo>/.testai-worktrees/<name>/` | Matches Claude Code's `.claude/worktrees/`. The 7-day volume TTL is plenty for a run. |
| Q3 | `testai/session-<id>` and `testai/sa-<id>` | Greppable, no collision with user branches. |
| Q4 | Per-subagent draft PR (via existing `commit_and_open_pr` tool) | Matches 2026 production. Tier-1 auto-merges; tier-2/3 awaits review. |
| Q5 | Per-session worktree = orchestrator's scratch space | Subagents branch off it. |
| Q6 | Auto-remove after subagent completes | Branch is on origin (the PR is its evidence); the worktree's only job was to host the commits. |

## Implementation

### New file: `backend/harness/services/worktree_manager.py` (~470 lines)

Ported from `reference/OpenHarness/src/openharness/swarm/worktree.py:135`
with TestAI-specific adjustments:

- **Git-runner abstraction**: `WorktreeManager` takes a
  `git_runner: GitRunner` callable. Production wires
  `sandbox_git_runner(env)` (runs `git` inside the Docker
  container); tests use `local_git_runner` (runs on the host).
  This is the seam that makes the manager work both in
  integration tests and in the live orchestrator.
- **Branch naming** per C01 Q3: `testai/session-<id>` and
  `testai/sa-<id>`.
- **Per-slug concurrency lock**: a per-(repo, slug) `asyncio.Lock`
  serializes concurrent `create_worktree` calls for the same
  slug. Without this, two coroutines see the worktree path as
  absent, both invoke `git worktree add`, and one of them loses
  with "fatal: <path> already exists".
- **Symlink common cache dirs** (OpenHarness pattern): `node_modules`,
  `.venv`, `__pycache__`, `.tox` are symlinked from the main repo
  into the worktree so per-worktree disk usage stays bounded.
- **Idempotent resume**: if the worktree directory exists and is
  a valid git worktree, return its info without re-running
  `git worktree add`. This makes the orchestrator's bootstrap call
  safe to repeat across runs.
- **Defensive fallback**: if `git worktree remove` fails (e.g.
  the main repo path is gone), fall back to `rm -rf` on the
  worktree directory, then `git worktree prune` to clean git's
  internal state.

### Modified: `backend/harness/orchestrator.py`

Added per-session worktree creation in the bootstrap of
`run_single` (after the KG index, so the worktree's branch is
based on the indexed HEAD):

```python
wt_manager = WorktreeManager()
session_worktree_info = await wt_manager.create_worktree(
    Path(repo_path) if repo_path else Path("/workspace/repo"),
    session_slug(session_id),
    branch=session_branch(session_id),
)
```

Best-effort: errors are logged at WARNING, the orchestrator still
runs without isolation. The KG continues to be per-repo, not
per-worktree (the worktree just gives a separate branch).

### Modified: `backend/harness/tools/delegate_task.py`

In `_run_single_enhanced`:

1. **Per-subagent worktree creation** (after the child is
   constructed, before tool dispatch):

   ```python
   wt_manager = WorktreeManager()
   subagent_worktree_info = await wt_manager.create_worktree(
       Path("/workspace/repo"),
       subagent_slug(subagent_id),
       branch=subagent_branch(subagent_id),
       agent_id=subagent_id,
   )
   ```

2. **Auto-remove in `finally`** (per C01 Q6):

   ```python
   removed = await wt_manager.remove_worktree(
       Path("/workspace/repo"),
       subagent_slug(subagent_id),
   )
   ```

Best-effort both ways. The existing `commit_and_open_pr` tool
remains the path for opening draft PRs — it runs inside the
sandbox and operates on whatever the agent's current working
directory is. The worktree just isolates the file writes; the
PR creation is unchanged.

### Future wiring (deferred)

The full C01 spec also covers:

- **Sandbox runner integration** (per-session worktree is created
  via the local git runner today; production would use
  `sandbox_git_runner(env)` once the sandbox env is reachable
  from the orchestrator).
- **`create_worker_env` worktree-aware path** (per Q1, the worker
  sandbox should use the subagent's worktree, not the shared
  `/workspace/repo`).
- **Auto PR-open at subagent completion** (today the agent has
  to call `commit_and_open_pr` itself; C01 future deepening can
  auto-call it when the subagent writes any file to the worktree).
- **PR base branch** (always `main`? the branch the orchestrator
  cloned from? configurable?).
- **`.worktreeinclude`** (Claude Code's pattern for copying
  gitignored env files like `.env`).
- **Worktree lock** (`git worktree lock` while running) to
  prevent a stuck child being reaped mid-work.

These are tracked in the open-questions section of the C01
decision tree.

## Behavior change (observable)

For a 30-minute orchestration with 5 parallel subagents on a
10k-file repo:

| | Before C01 | After C01 |
|---|---|---|
| Per-subagent disk usage | full repo (10k files) | full repo + 4 symlinks (node_modules, .venv, etc) |
| Per-subagent branch | shared (race risk) | isolated `testai/sa-<id>` |
| Orchestrator's branch | shared with user | `testai/session-<id>` (sandboxed) |
| Subagent cleanup | none (worktree dir accumulates) | auto-remove after subagent finishes |
| PR-based merge | manual | `commit_and_open_pr` opens draft PR (existing tool) |

## Failure modes

| Failure | Detection | Behavior |
|---|---|---|
| `git` not on PATH | `_run_git` returns non-zero | `WorktreeError` raised with stderr |
| Repo isn't a git repo | `git rev-parse --git-dir` fails | `WorktreeError` raised |
| `git worktree add` fails (e.g. branch conflict) | non-zero exit | `WorktreeError` raised |
| Concurrent `create_worktree` for same slug | per-slug lock | Second call resumes instead of duplicating |
| `git worktree remove` fails (e.g. dirty tree) | non-zero exit | Falls back to `rm -rf`, then `git worktree prune` |
| Worktree init fails in orchestrator | caught in `try/except` | Logged at WARNING, orchestrator continues without isolation |
| Worktree init fails in delegate_task | caught in `try/except` | Logged at DEBUG, subagent runs without isolation |
| Symlink creation fails (e.g. Windows mount) | `OSError` | Logged at DEBUG, symlink skipped (non-fatal) |
