Problem

Five competing data-fetching patterns with no contract for how a page should fetch data. Each pattern handles loading state, error state, retry, caching, and refresh differently. The SSE connection for Chat is 125 lines of inline EventSource wiring that can't be reused by any other page. useEffect + api.get() pages silently swallow errors (.catch(() => {})). Pages like Dashboard mix useQuery, Zustand store fetches, and direct api calls — three patterns in one page. The Wolf Tech decision matrix recommends: queries (polling 30s+), SSE (unidirectional streaming), WebSocket (bidirectional). This codebase uses all three but without the discipline of choosing the right one per data type and sticking to it.

Solution

Declare a frontend data layer contract: Queries (infrequent data) → useQuery with global defaults (staleTime: 30s, retry: 3, refetchOnWindowFocus: true). Streams (real-time events) → one useEventSource hook with standard reconnect, ring buffer, pause/resume — no inline EventSource wiring. UI state (client-only) → Zustand stores without API calls mixed in. Eliminate useEffect + api.get() + setState entirely — migrate those 5 pages to useQuery. The StormHub harness uses @ai-sdk/react useChat as its single streaming primitive. The a0 harness uses one global store. OpenHands uses one useWebSocket hook. The pattern is one per data-class, not five.
