# C02 — Agent Teams

**Date**: 2026-06-21
**Status**: Implemented (MVP)
**Decision tree**: `docs/2026-06-21-architecture-decision-tree.md#c02`

## Problem

TestAI's subagent model today treats every `delegate_task` call as
isolated. Subagents can see the kanban board (passive
observability) but can't message each other, share state, or
coordinate. For multi-step work like "explore → fix → verify", the
parent orchestrator has to ferry every message between subagents,
which (a) costs context budget and (b) makes the orchestrator a
bottleneck.

Claude Code v2.1.x (May 2026) added **Agent Teams** — multiple
coordinated sessions with a shared task list + inter-agent
messaging, managed by a lead. C02 brings this to TestAI on top of
C01's per-subagent worktrees.

## Decision (locked, 6 questions)

| Q | Decision | Rationale |
|---|----------|-----------|
| Q1 | Pure coordination (no team-aggregation) | Per-subagent PRs (C01) are the integration surface; aggregation would re-introduce the team-PR pattern. |
| Q2 | Dynamic via `team_create` tool | The LLM has the context to know when a team is needed. |
| Q3 | Lead's toolset = full `team_*` surface | Lead coordinates; needs create/messaging/dissolve. |
| Q4 | Member's toolset = read + reply | Members can ask questions but don't manage the team. |
| Q5 | Hybrid lifecycle (explicit OR auto) | Lead can dissolve; system auto-dissolves when all members done. |
| Q6 | Postgres storage | Cross-process visibility; matches TestAI's existing patterns. OpenHarness uses filesystem (single-process CLI) — doesn't fit. |

## Implementation

### Modified: `backend/harness/memory/schema/migrations.sql`

Three new tables (ADR-0014):

```sql
CREATE TABLE teams (
    team_id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    lead_subagent_id TEXT,
    lead_session_id TEXT,
    goal TEXT DEFAULT '',
    status TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'dissolved'
    created_at TIMESTAMPTZ DEFAULT NOW(),
    dissolved_at TIMESTAMPTZ,
    config JSONB DEFAULT '{}'::jsonb
);
CREATE TABLE team_members (
    team_id TEXT REFERENCES teams(team_id) ON DELETE CASCADE,
    subagent_id TEXT NOT NULL,
    role TEXT NOT NULL DEFAULT 'member',  -- 'lead' | 'member'
    role_name TEXT DEFAULT '',
    status TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'done' | 'failed'
    joined_at TIMESTAMPTZ DEFAULT NOW(),
    left_at TIMESTAMPTZ,
    PRIMARY KEY (team_id, subagent_id)
);
CREATE TABLE team_messages (
    id BIGSERIAL PRIMARY KEY,
    team_id TEXT REFERENCES teams(team_id) ON DELETE CASCADE,
    from_subagent_id TEXT,
    to_subagent_id TEXT,         -- NULL = broadcast
    content TEXT NOT NULL,
    kind TEXT NOT NULL DEFAULT 'message',  -- 'message' | 'status' | 'system'
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

Indexes on `teams.status`, `team_members.subagent_id`,
`team_messages(team_id, created_at)`, and a partial index on
`team_messages(team_id, to_subagent_id)` for the per-recipient
view.

### New file: `backend/harness/services/team_service.py` (~620 lines)

The `TeamService` is a thin data-access layer over the three
tables. The full CRUD surface:

- `create_team` / `get_team` / `get_team_including_dissolved` / `list_teams`
- `dissolve` (idempotent; posts a system message)
- `add_member` / `remove_member` / `update_member_status` / `list_members` / `get_member`
- `send_message` / `list_messages` (with per-recipient filtering)
- `cleanup_completed` (Q5: auto-dissolve teams where all members are done)

Data classes (`Team`, `TeamMember`, `TeamMessage`) are frozen
dataclasses with `to_dict()` for JSON serialization.

Errors: `TeamNotFoundError`, `TeamDissolvedError`,
`MemberNotFoundError`, base `TeamError`.

Validation: `validate_team_name` rejects empty / too-long /
disallowed-character names (max 100 chars, allows letters,
digits, spaces, `._:-`).

### New file: `backend/harness/tools/team_tools.py` (~330 lines)

Six `BaseTool` subclasses registered in two toolsets:

**Lead (full surface)** — `team_lead` toolset:
- `team_create` — Q2: create a new team
- `team_message` — Q3: post a message (broadcast if `to_subagent_id` is omitted)
- `team_list_messages` — Q3: read team's thread
- `team_list_members` — Q3: read members
- `team_member_progress` — Q3: members' statuses + last activity
- `team_dissolve` — Q3 + Q5: explicit dissolve

**Member (read + reply)** — `team_shared` toolset:
- `team_message`
- `team_list_messages`
- `team_list_members`

The "lead-only" tools check `is_team_lead` at the toolset level.
For the MVP, we don't enforce it at the tool level — a sufficiently
smart member could call `team_dissolve` if it's in their
toolset. The `toolsets.py` filter is the canonical enforcement
point.

### Modified: `backend/harness/tools/delegate_task.py`

Added `team_id: str` and `is_team_lead: bool` parameters to the
`delegate_task` tool. When `team_id` is set:

1. **Before spawn**: register the subagent with the team via
   `TeamService.add_member(team_id, subagent_id, role=...)`.
   - If the team doesn't exist, return immediately with
     `status="error", error="team_not_found: ..."`.
   - Membership registration is best-effort: if the DB is
     unavailable, the subagent still runs without team context.
2. **During run**: the subagent runs as normal. Its `team_*`
   tools (from the toolset) give it messaging + read access.
3. **After completion (finally block)**: mark the member as
   `done` (or `failed` if the subagent errored). This lets
   `TeamService.cleanup_completed` auto-dissolve the team when
   every member finishes.

The `team_id` and `is_team_lead` parameters also thread through
`_run_single_enhanced` so the same logic applies to sync,
fan-out, and background modes.

## Behavior change (observable)

For a 5-subagent team doing "explore → fix → verify":

| | Before C02 | After C02 |
|---|---|---|
| Inter-subagent messaging | orchestrator ferries every message | subagents message directly via `team_message` |
| Shared state | none | team thread visible to all members |
| Coordination cost | orchestrator's context budget | small (tool calls only) |
| Lifecycle | manual | explicit dissolve OR auto-dissolve on all-done |

## Failure modes

| Failure | Detection | Behavior |
|---|---|---|
| `team_id` doesn't exist | `TeamNotFoundError` at registration | `SubagentResult(status="error")` returned before spawn |
| DB unavailable at registration | caught in `try/except` | Logged at DEBUG, subagent runs without team context |
| Member tries to message non-member | `MemberNotFoundError` | Tool returns `member_not_found` error |
| `send_message` content too long | truncated to 10K chars + `…` | Acceptable — long messages should be split |
| `get_team` for dissolved team | filtered at query level | `TeamNotFoundError` raised |
| `dissolve` already-dissolved | idempotent | Returns `False` (newly_dissolved=false) |
| Lead dies before dissolving | members never see `done` | Team never auto-dissolves; manual intervention needed |

## Open follow-ups

1. **Auto-sweeper cron** — `cleanup_completed` is exposed but
   no caller invokes it on a schedule. The natural place is
   a `team_sweeper` background task that runs every 60s and
   calls `cleanup_completed` (mirrors the kanban review agent
   pattern at `services/kanban_service.py:553-615`).
2. **Team creation via `delegate_task` directly** — currently
   the coordinator calls `team_create`, then a separate
   `delegate_task(team_id=...)` for each member. A more
   ergonomic API would be `delegate_task(team_id=..., role="lead",
   members=[{...}])` that creates + spawns in one call.
3. **Nested teams** — for v1, single-level only (a lead can't
   spawn its own team).
4. **Cross-team messaging** — for v1, no (a member of team A
   can't message a member of team B).
5. **Lead succession** — if the lead's process dies, the team
   is left in a weird state. Could auto-elect a new lead or
   auto-dissolve.
6. **Team cost budgets** — currently each member has its own
   budget; could share a team budget.
7. **Deeper message filtering** — currently `team_list_messages`
   returns broadcasts + direct messages + the member's own
   outbound. A member might want to see only messages addressed
   to them (no broadcasts).
