## Goal

Promote TestAI's OTel spans to a first-class observability output. The plumbing exists (`harness/trace.py::OTelTraceHandler` is wired via `api/state.py::trace_handler` and conforms to OTel GenAI semconv), but the surface is **partial**:

- ✓ LLM chat spans (`gen_ai.operation.name=chat`, `gen_ai.request.model`, `gen_ai.usage.*`)
- ✓ Tool execution spans (`gen_ai.tool.name`, `testai.tool.success`)
- ✓ Agent lifecycle spans (`gen_ai.agent.id`, `testai.agent.parent_id`)
- ✗ Subagent invocation spans (`testai.subagent.invoke` — F04 gap)
- ✗ Kanban transition spans (`testai.kanban.transition` — F04 gap)
- ✗ Budget throttle spans (`testai.budget.throttled` — F04 gap, extends C07)
- ✗ Opt-in toggle (`OTEL_ENABLED` env var — currently always-on)
- ✗ Status endpoint + UI

F04 closes all four gaps.

## Why now (triggers)

- **Report flagged this as the F-series priority** — 2 weeks, leverages C03's event pipeline, unblocks any future observability/audit work.
- **The 2026 OTel GenAI spec graduated CNCF** in May 2026 — `gen_ai.*` is now the canonical attribute namespace for any AI agent observability story. TestAI already conforms; F04 completes the surface.

## Why these specific attrs

- **`testai.subagent.id`** — the subagent's UUID, used for `gen_ai.agent.id` association in tool/llm child spans (downstream fan-out)
- **`testai.subagent.role`** — e.g. "test-writer", "code-reviewer"; matches the existing `Role` YAML vocabulary
- **`testai.subagent.depth`** — recursion depth (0 = root agent, 1 = direct subagent, etc.)
- **`testai.kanban.transition`** — the kind of transition (`completed` / `failed` / `blocked`); enables OTel queries like "all failed kanban transitions in the last hour"
- **`testai.budget.throttle_step`** — 0-4; the existing C07 ladder. Surfaced as an OTel attribute so external observability dashboards can chart the throttle step over time

## Opt-in toggle

Today OTel is initialized at import time in `harness/trace.py`. If the SDK isn't installed or the endpoint isn't reachable, it silently no-ops (fail-open). F04 adds an **explicit opt-in** so deployments that don't want OTel don't pay the import cost or the gRPC export overhead.

```python
# harness/trace.py
def _init_otel():
    global OTEL_AVAILABLE, _tracer
    if not _is_otel_enabled():
        OTEL_AVAILABLE = False
        return
    # ... existing init code ...

def _is_otel_enabled() -> bool:
    return os.environ.get("OTEL_ENABLED", "false").lower() in ("true", "1", "yes")
```

The env var `OTEL_ENABLED=true` is the on switch. The endpoint, service name, and other settings are unchanged. **Default is off** — deployments that want OTel explicitly set the env var.

The status endpoint (below) reports the current state so the operator can see whether OTel is on, off, or failed to initialize.

## Status endpoint

`GET /api/observability/status` returns:

```json
{
  "enabled": true,
  "available": true,
  "endpoint": "http://localhost:4317",
  "service_name": "testai-harness",
  "service_version": "1.0.0",
  "span_counts_last_5_min": {
    "chat": 142,
    "execute_tool": 87,
    "subagent_invoke": 12,
    "kanban_transition": 38,
    "kanban_board": 4,
    "budget_throttle": 1,
    "agent_run": 3,
    "agent_round": 21
  },
  "last_span_at": "2026-06-22T12:45:33.142Z"
}
```

`available: false` when the OTel SDK is missing or the exporter failed to initialize. `enabled: true, available: false` means "you asked for OTel but it can't run" — operator needs to fix the SDK or endpoint.

The span counts come from a small in-memory counter on `OTelTraceHandler` (a `dict[str, int]` keyed by operation name, with a 5-minute rolling window). Reset on every process restart; no DB persistence (per OTel norms — the exporter ships to the backend).

## UI: Observability status

A new section on the dashboard (or a dedicated `/observability` page). Per the design-taste-frontend skill:

- **Off-Black base** with a single accent (emerald-500 for "ok", rose-500 for "error")
- **Asymmetric layout** — endpoint on the left, span counts on the right
- **No emojis** — icons via Phosphor or Radix
- **Monospace numbers** for the counts (`font-mono tabular-nums`)
- **Per-frontend Skill §9.C** — bar-chart micro-animation when a count is updating

The page has 3 regions:
1. **Top: status bar** — enabled/disabled + endpoint + service version
2. **Middle: live span counts** — grid of count cards (one per operation type), updated every 5s via a small polling hook
3. **Bottom: recent spans** (optional) — last 10 spans with their name + duration + attributes (truncated). Polled every 5s.

For C07/F04 effort, we ship the status bar + live counts (regions 1+2). Region 3 (recent spans) is deferred — it requires more API surface (in-memory span ring buffer + GET /api/observability/spans).

## Design decisions (grilled 2026-06-22)

1. **Add spans, don't refactor** — `OTelTraceHandler` already follows the right pattern; F04 just adds 4 more `elif` branches.
2. **Custom attrs under `testai.*`** — `gen_ai.*` is reserved for the OTel spec. Per the [OTel naming guideline](https://opentelemetry.io/docs/specs/semconv/general/attribute-naming/): "Custom attributes used MUST have a non-empty namespace."
3. **Opt-in via `OTEL_ENABLED` env var, default off** — current code is always-on. F04 makes it explicit so deployments that don't want OTel don't pay the cost. Status endpoint confirms current state.
4. **Fail-open semantics preserved** — if the SDK is missing or the endpoint is unreachable, spans are dropped silently. The status endpoint reports the failure.
5. **`testai.kanban.transition` is a span kind, not an attribute** — every kanban event is its own span so the OTel timeline shows the per-task cadence.
6. **`testai.budget.throttled` reuses the C07 `BudgetThrottled` event** — no new emit path; OTel subscribes to the existing EventBus event. One source of truth.
7. **Status endpoint is read-only** — no `POST /api/observability/toggle`. Toggling happens via env var + process restart, matching the OTel community's preferred model.
8. **UI: status bar + live counts now, recent spans later** — region 3 needs a ring buffer that can be heavy; defer to keep the F04 effort at 2 weeks.
9. **No exporter config UI** — operators configure OTLP_ENDPOINT + OTEL_ENABLED via env vars + config file, not the dashboard. The dashboard is observability, not config.
10. **`subagent_invoke` operation name** (not `subagent.run`) — matches the test file convention `testai.subagent.invoke` from the F04 spec. The OTel `gen_ai.operation.name` value should be a verb-noun, so `subagent_invoke` reads as "this span describes a subagent invocation".

## What this fixes

- **"Where did the time go?"** — full timeline view of subagent spawn, kanban progress, LLM calls, tool calls, all on one OTel timeline
- **Cost attribution** — `testai.budget.throttled` spans make the throttle ladder observable from external dashboards (Datadog, Honeycomb, Grafana Tempo, etc.)
- **Subagent debugging** — `testai.subagent.*` attrs let operators ask "which subagent role failed the most this week?" without parsing logs
- **OTel parity with Claude Code v2.1.x** — TestAI now exposes the same surface Claude Code does (and more: subagent + kanban are TestAI-specific extensions)

---

## What this does NOT fix (deferred)

- **Recent spans endpoint (region 3 of the UI)** — needs a ring buffer + pagination; future sprint
- **Per-tool attribute coverage** — we set `gen_ai.tool.name` but not the args (no PII; configurable via env var in a future sprint)
- **Span links between parent + child subagent** — the `traceparent` header propagation is a future sprint
- **OTel metrics** (counters, histograms) — F04 is spans-only. Future sprint adds `gen_ai.client.token.usage` histogram etc.
- **OTel logs** — out of scope for F04
- **Exemplar sampling** — we emit every span; production at scale would want tail-based sampling

## Wire-level example

```bash
# Enable OTel (off by default in F04).
export OTEL_ENABLED=true
export OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317
export OTEL_SERVICE_NAME=testai-harness
export OTEL_SERVICE_VERSION=1.0.0

# Start the backend.
python -m uvicorn api.main:app
# → observability: OpenTelemetry sink initialized (OTLP gRPC → http://otel-collector:4317)

# Run a job.
# → subagent.spawned → testai.subagent.invoke span ("subagent_invoke test-writer" + role=...)
# → llm:start → chat span ("chat gpt-4o" + tokens=...)
# → tool:start → execute_tool span ("execute_tool bash" + tool.name=...)
# → board.task.completed → kanban_transition span ("kanban_transition" + transition=completed)
# → board.completed → kanban_board span ("kanban_board" + task_count=5)
# → budget.throttled → budget_throttle span ("budget_throttle" + step=1 + hitl_active=true)

# View in Honeycomb / Datadog / Grafana Tempo.
# → query: "all testai.budget.throttle_step >= 3 in the last hour" returns the 4-step climb
# → query: "p95 of testai.subagent.duration_s grouped by testai.subagent.role" returns per-role latency
# → query: "count of testai.kanban.transition=failed grouped by testai.kanban.task_role" returns failure rate by role
```
