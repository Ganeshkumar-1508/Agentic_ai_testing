# C05 — A2A Wire Protocol

**Date**: 2026-06-21
**Status**: implemented
**Owner**: TestAI Harness
**Spec**: [A2A Protocol v1.0](https://a2a-protocol.org/latest/) (Linux Foundation, June 2025)

---

## Goal

Expose TestAI's orchestrator as a **standards-compliant A2A Server** so any A2A client (Google ADK, LangGraph agents, OpenAI Agents SDK with the A2A adapter, custom UIs) can submit jobs, stream events, poll status, and cancel/pause runs without knowing the TestAI HTTP API. A2A + MCP is the 2026 cross-vendor agent interoperability stack — TestAI is MCP-native, and C05 is the A2A twin.

The A2A spec is **HTTP + JSON-RPC 2.0 + SSE**. TestAI is **HTTP + REST + SSE**. C05 is a thin shim that maps one to the other.

---

## Why now (triggers)

- **Industry alignment** — A2A v1.0 launched June 2025; major vendors (Google, AWS, Microsoft, Cisco, IBM, Salesforce) ship A2A clients. TestAI's value prop ("agent harness for testing") gets amplified if external agents can call us.
- **2026 architecture call** — C05 was identified in the 2026-06-21 architecture review as the **one deepening that exposes the platform to the rest of the agent ecosystem** without changing internal APIs.
- **C08 is now stable** — JobSpec canonicalization (Q4-Q7 step 2) gives us a single, typed submission shape to map onto A2A's `SendMessage` / Parts.
- **Streaming already works** — the C08 SSE event pipeline (`/api/events/{session_id}` + `EventSourceSink.subscribe`) is exactly the substrate A2A's `SendStreamingMessage` wants. C05 is plumbing, not plumbing-from-scratch.

---

## What C05 is NOT

- **Not an A2A client** — we expose A2A Server, not consume A2A from peer agents. (Future: C05-b would add an A2A client tool the coordinator can use to call external agents.)
- **Not a gRPC server** — A2A v1.0 is JSON-RPC 2.0 over HTTP(S); gRPC and REST transports are spec-defined but optional. We only ship the JSON-RPC binding.
- **Not a push notification server** — A2A v1.0 supports webhooks (`tasks/pushNotificationConfig/*`). We skip these in C05; SSE is sufficient for our 2026 needs. Defer to C05-b if the user asks for disconnected-client support.
- **Not a replacement for the chat or REST surface** — A2A is a third submission source, alongside chat (`POST /api/jobs` via `submit_job` tool) and webhooks. All three funnel into the same `JobSpec` pipeline.

## Architecture: thin shim over `/api/jobs`

```
┌────────────────┐                 ┌─────────────────────────────────────────────┐
│  A2A Client    │  JSON-RPC 2.0   │  TestAI Backend (A2A Server)               │
│  (any vendor)  │ ───────────────►│                                             │
│                │                 │  GET /.well-known/agent.json                │
│                │                 │      → static Agent Card (name, skills,    │
│                │                 │        url, version, capabilities)         │
│                │                 │                                             │
│                │  SendMessage    │  POST /a2a/jsonrpc  (method=SendMessage)   │
│                │ ───────────────►│      → build JobSpec from Message+Parts    │
│                │                 │      → submit_job_to_orchestrator (C08)    │
│                │                 │      → return Task with status             │
│                │                 │                                             │
│                │  SendStream     │  POST /a2a/jsonrpc  (method=SendStream)    │
│                │ ───────────────►│      → submit, then SSE bridge from        │
│                │  ◄─────SSE──────│        EventSourceSink.subscribe           │
│                │                 │      → stream task-status-update +          │
│                │                 │        artifact chunks in A2A shape        │
│                │                 │                                             │
│                │  GetTask        │  POST /a2a/jsonrpc  (method=GetTask)       │
│                │ ───────────────►│      → load JobSpecRecord + comments       │
│                │                 │      → return Task with full status        │
│                │                 │                                             │
│                │  CancelTask     │  POST /a2a/jsonrpc  (method=CancelTask)    │
│                │ ───────────────►│      → JobSpecStore.cancel → orchestrator  │
│                │                 │        cancel_watcher → task cancelled     │
└────────────────┘                 └─────────────────────────────────────────────┘
```

The whole A2A implementation is **~700 lines** in `backend/harness/a2a/` because every state transition routes through existing C08 seams.

## A2A v1.0 spec conformance

We implement the **5 most-used methods** from the [A2A v1.0 spec](https://a2a-protocol.org/latest/) and **skip the optional push-notification surface**.

### Methods (JSON-RPC 2.0 over `POST /a2a/jsonrpc`)

| A2A method              | Maps to                                       | Implementation                  |
|-------------------------|-----------------------------------------------|---------------------------------|
| `SendMessage`           | `POST /api/jobs`                              | `a2a/server.py::send_message`   |
| `SendStreamingMessage`  | `POST /api/jobs` + SSE bridge                 | `a2a/server.py::send_streaming_message` + `a2a/stream.py` |
| `GetTask`               | `GET /api/jobs/{spec_id}`                     | `a2a/server.py::get_task`       |
| `CancelTask`            | `POST /api/jobs/{spec_id}/cancel`             | `a2a/server.py::cancel_task`    |
| ~~`tasks/resubscribe`~~ | Skipped (SSE reconnection is in SendStream)   | —                               |
| ~~Push notifications~~  | Deferred to C05-b                             | —                               |

**Method naming**: A2A v1.0 uses **CamelCase** JSON-RPC methods (`SendMessage`, not `message/send`). Early v0.x used slash-namespaced methods; v1.0 is the merged standard. We support CamelCase only.

### Task lifecycle

| A2A `TaskState`             | TestAI `JobStatus` |
|-----------------------------|--------------------|
| `TASK_STATE_SUBMITTED`      | `pending`, `submitted`, `queued` |
| `TASK_STATE_WORKING`        | `running`         |
| `TASK_STATE_COMPLETED`      | `completed`       |
| `TASK_STATE_FAILED`         | `failed`          |
| `TASK_STATE_CANCELED`       | `cancelled`       |
| `TASK_STATE_INPUT_REQUIRED` | (n/a — TestAI has no input-required pause) |
| `TASK_STATE_AUTH_REQUIRED`  | (n/a — TestAI has no auth-required pause) |
| `TASK_STATE_REJECTED`       | (n/a — Tier-2 review maps to `paused`) |
| `TASK_STATE_WORKING_COMPLETED` | (n/a — not a hybrid agent)   |

TestAI is a **Task-generating Agent**, not a Hybrid Agent. We never return `Message` for a job submission — we always return a `Task`. The spec describes this as the simpler model: "Avoids deciding between `Task` versus `Message`". We adopt it.

### Part → JobSpec mapping

A2A `Message.parts[]` is a flexible content container. We support all 4 kinds:

| A2A `Part.kind`   | `Part` field    | Maps to `JobSpec` |
|-------------------|-----------------|-------------------|
| `text`            | `text: str`     | Concatenated into `JobSpec.prompt` (with `\n\n` separator between multiple text parts) |
| `data`            | `data: object`  | Merged into `JobSpec.context` (with `part.metadata.title` as key prefix when present) |
| `url`             | `url: str`      | First `url` part → `JobSpec.repo_url` |
| `raw` (file)      | `raw: bytes`    | Ignored in v1 (no file ingestion path) — return error if seen |

This matches the A2A tutorial example verbatim. We don't invent new content kinds.

### Artifact → JobOutput mapping

A2A `Artifact` is "tangible output". We map `JobOutput` → 1-3 Artifacts:

1. **Test files artifact** — every committed/modified test path from the orchestrator's kanban board becomes one `Artifact.parts[]` with a `text` part containing the path list and a `data` part containing `{count, paths}`.
2. **PR artifact** — if `JobOutput.pr_url` is set, one Artifact with `description: "Pull request"` and a `data` part `{url, title, branch}`.
3. **Summary artifact** — the `JobOutput.summary` text becomes a `text` Artifact named "summary".

All three are emitted on terminal-state Task responses. The streaming variant emits **delta** artifacts incrementally (A2A v1.0 explicitly allows `artifact-update` events with chunked parts).

### SSE event mapping (streaming variant)

The bridge from `EventSourceSink.subscribe` to A2A's `SendStreamingMessage` event stream:

| `StreamEvent.type`         | A2A `SendStreamingMessageResponse.result` |
|----------------------------|------------------------------------------|
| `connected` (first frame)  | (no-op, internal keepalive)              |
| `job.submitted`            | `Task` with `status.state=SUBMITTED`     |
| `board.task.completed`     | `TaskStatusUpdateEvent(state=WORKING)` + chunked `Artifact` of test files |
| `subagent.completed`       | `TaskArtifactUpdateEvent` (subagent output → delta Artifact) |
| `run.completed`            | `TaskStatusUpdateEvent(state=COMPLETED)` + final `Task` |
| `run.failed`               | `TaskStatusUpdateEvent(state=FAILED)` |
| `ping` (keepalive)         | (no-op, internal)                       |

A2A v1.0 SSE envelopes:
```
event: task-status-update
data: {"jsonrpc":"2.0","id":"req-001","result":{"taskId":"...","contextId":"...","status":{"state":"TASK_STATE_WORKING","message":"..."}}}

event: task-artifact-update
data: {"jsonrpc":"2.0","id":"req-001","result":{"taskId":"...","artifact":{"artifactId":"...","parts":[...]}}}

event: message
data: {"jsonrpc":"2.0","id":"req-001","result":{"kind":"status-update",...}}
```

We send **two SSE frames per event** to be conservative: the typed-event frame (for clients that filter on `event:`) and a generic `message` frame (for clients that use `onmessage` only). The browser `EventSource` API routes `event:` frames to `addEventListener(type, ...)`; `message` frames go to `onmessage`.

Wired into `api/main.py`:
```python
from harness.a2a.server import a2a_router
...
app.include_router(a2a_router)
```

The A2A router has its own `APIRouter` and its own URL prefix (`/a2a`) so the A2A surface is **self-contained** — a vendor client connecting to `https://testai.example.com/a2a/jsonrpc` never sees `/api/jobs` or any other internal route. This matches the A2A spec's discovery-via-Agent-Card model: clients find us, they read our Agent Card, and the Agent Card is the only entry point they need.

## Auth: bearer JWT only

A2A v1.0 defers to standard web security. We use **HS256 bearer JWT** validated by a FastAPI dependency on the A2A router. The same token that protects the chat surface (already in production) is reused. The Agent Card's `authentication.schemes` lists `["bearer"]`.

For local dev, the A2A router is mounted **unauthenticated** if `TESTAI_A2A_AUTH_DISABLED=1`. This lets the A2A Playground ([a2a-protocol.org/playground](https://a2a-protocol.org/latest/tutorials/python/1-introduction/)) hit us without ceremony.

## Mapping: JobStatus → A2A TaskState

The single most important decision in C05 is the lifecycle mapping. We use a function `a2a_job_status_to_task_state(status: str) -> str` that returns the A2A `TASK_STATE_*` enum value for any `JobStatus` string. The mapping is **lossy in the input-required / auth-required direction** (TestAI has no such state) but **lossless in the terminal direction**.

```python
JOB_STATUS_TO_A2A_STATE: dict[str, str] = {
    "pending":   "TASK_STATE_SUBMITTED",
    "queued":    "TASK_STATE_SUBMITTED",
    "submitted": "TASK_STATE_SUBMITTED",
    "running":   "TASK_STATE_WORKING",
    "completed": "TASK_STATE_COMPLETED",
    "failed":    "TASK_STATE_FAILED",
    "cancelled": "TASK_STATE_CANCELED",
    "paused":    "TASK_STATE_INPUT_REQUIRED",  # A2A has no "paused"; we surface as input-required
}
```

**The `paused → input-required` mapping is the only one that's not 1:1.** TestAI's `paused` state means "user requested pause, will not resume until user re-submits". A2A's `input-required` means "agent needs more input to continue". They're operationally identical from the client's POV: "the agent stopped and is waiting on you." We map `paused → input-required` so A2A clients see the spec-correct state. The A2A spec explicitly allows servers to add custom states via extensions, but a clean mapping is simpler and the semantic is close enough.

## Design decisions (grilled 2026-06-21)

1. **Thin shim, not a full A2A SDK** — we don't vendor `a2a-sdk`. We write ~700 lines of Pydantic models + FastAPI handlers. The `a2a-sdk` is for SDK consumers (clients); we're a server.
2. **One transport: JSON-RPC 2.0 over HTTP** — no gRPC, no REST binding. JSON-RPC is the canonical A2A binding; clients prefer it.
3. **CamelCase method names** — `SendMessage`, not `message/send`. v1.0 standard.
4. **5 methods, not 8** — skip `tasks/resubscribe` (SSE reconnect covers it) and `tasks/pushNotificationConfig/*` (defer to C05-b).
5. **Task-generating agent, not Hybrid** — we never return a `Message`; we always return a `Task`. Simpler model.
6. **Bearer JWT, same as chat** — no new auth surface. The A2A router reuses the chat's HS256 validation.
7. **`paused → input-required`** — closest A2A semantic. Document the choice in the Agent Card `description`.
8. **Self-contained `/a2a` prefix** — the A2A router is its own FastAPI router, not a re-skin of `/api/jobs`. Clean separation; clients see a single entry point.
9. **No tests for the optional parts** — `FilePart` (raw bytes) returns an error; we don't support file ingestion in C05.
10. **Agent Card is dynamic** — built at request time so the `url` reflects the actual deployment (env-driven), not a hardcoded string.

## What's deferred to C05-b

- `tasks/resubscribe` method (SSE reconnection, low priority since `SendStreamingMessage` already streams)
- `tasks/pushNotificationConfig/*` (webhook subscriptions, deferred until a customer asks)
- `FilePart` ingestion (raw bytes → repo upload, would need a file-storage surface)
- Auth extensions (OAuth 2.0, mTLS — currently we only do bearer JWT)
- A2A **client** tool the coordinator can use to call external agents (the spec is symmetric; a Client tool would be a separate tool registration)

## Wire-level examples

### Discovery

```http
GET /.well-known/agent.json HTTP/1.1
Host: testai.example.com
```

Response: Agent Card (above).

### Submit a job

```http
POST /a2a/jsonrpc HTTP/1.1
Content-Type: application/json

{
  "jsonrpc": "2.0",
  "id": "req-001",
  "method": "SendMessage",
  "params": {
    "message": {
      "role": "user",
      "messageId": "msg-001",
      "parts": [
        {"text": "Add Jest tests for the auth module in "},
        {"url": "https://github.com/acme/api", "metadata": {"label": "repo"}},
        {"text": ". Focus on the JWT validation paths."}
      ]
    }
  }
}
```

Response:
```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "result": {
    "task": {
      "id": "spec-abc-123",
      "contextId": "session-xyz-789",
      "status": {"state": "TASK_STATE_SUBMITTED"},
      "artifacts": []
    }
  }
}
```

### Stream events (SSE)

```http
POST /a2a/jsonrpc HTTP/1.1
Content-Type: application/json
Accept: text/event-stream

{
  "jsonrpc": "2.0",
  "id": "req-002",
  "method": "SendStreamingMessage",
  "params": {"message": { ... same as above ... }}
}
```

SSE stream:
```
event: task-status-update
data: {"jsonrpc":"2.0","id":"req-002","result":{"taskId":"spec-abc","status":{"state":"TASK_STATE_WORKING","message":"Sandbox ready"}}}

event: task-artifact-update
data: {"jsonrpc":"2.0","id":"req-002","result":{"taskId":"spec-abc","artifact":{"artifactId":"art-tests-1","name":"test_files","parts":[{"data":{"count":3,"paths":["tests/auth.test.ts","tests/jwt.test.ts","tests/oauth.test.ts"]}}]}}}

event: task-status-update

```
