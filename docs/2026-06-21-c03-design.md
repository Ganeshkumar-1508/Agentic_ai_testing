# C03 — Push-based board completion

**Date**: 2026-06-21
**Status**: Implemented
**Decision tree**: `docs/2026-06-21-architecture-decision-tree.md#c03`

## Problem

`OrchestratorEngine._wait_for_board` polled `cmd_orchestrate_monitor`
every 15 seconds for up to 5400s. The cost was:

- 1 DB query + 1 JSON parse every 15s per active run
- Worst-case 360 DB hits per run for nothing — the board is either
  done or stuck, the poll doesn't help disambiguate.
- Every subagent's "I'm done!" had to wait for the next poll cycle
  to be visible to the orchestrator.
- For a 1-hour run with 100 tasks, the orchestrator did ~240 polls
  and missed every individual task completion by 0-15s.

## Decision (locked, 4 questions)

| Q | Decision | Rationale |
|---|----------|-----------|
| Q1 | Subscribed events: `board.completed` + `board.failed` + `subagent.completed` | One terminal per board (board.completed/failed) + one observation per subagent (subagent.completed for failure cascade). No need for tool/round events — too chatty. |
| Q2 | Subscription mechanism: `asyncio.Event` via existing `EventSourceSink.subscribe(session_id)` | Already there, already used by SSE consumers. Per-session queue (queue_max=1024) handles the load. No new infrastructure. |
| Q3 | Idempotency key: `(subagent_id, status)` tuple set | Hermes' loop-guard pattern. Catches retried events. Failed subagent doesn't terminate — board.failed will. |
| Q4 | Fallback: poll `cmd_orchestrate_monitor` every 15s after 60s of no events | Hybrid. Push works in-process; cross-process / dropped events fall back to poll within 60s. Catches cases where the EventBus doesn't deliver (e.g. worker in a different process). |

## Implementation

### New file: `backend/harness/services/board_waiter.py` (~350 lines)

The `BoardWaiter` class:

```python
async def wait(self) -> BoardWaitResult:
    if self._sink is None:
        return await self._wait_pure_poll()  # backwards-compat
    queue = self._sink.subscribe(self._session_id)
    try:
        return await self._wait_with_push(queue)
    finally:
        self._sink.unsubscribe(self._session_id, queue)
```

`BoardWaitResult` is a frozen dataclass:

```python
@dataclass(frozen=True)
class BoardWaitResult:
    success: bool
    status: Literal["completed", "stalled", "blocked", "failed", "timed_out"]
    board_id: str
    tasks: list[dict] = []
    stalled_tasks: list[dict] = []
    blocked_tasks: list[dict] = []
    elapsed_seconds: float = 0.0
    method: Literal["push", "poll", "timeout"] = "timeout"
    events_received: int = 0
    last_event_at: float = 0.0
```

### Modifications

**`backend/harness/services/kanban_service.py`** — `complete_task` /
`block_task` / `claim_task` (auto-blocked) / `review_task` (approve) /
`update_task` (terminal column) all call the new
`_emit_board_completion_if_done(board_id)` helper. The helper does a
single aggregate query (`COUNT(*) FILTER (WHERE ...)`) and, if every
task is in a terminal column, emits `board.completed` (all done) or
`board.failed` (any blocked) with the full task list and a routing
key derived from `board.config.session_id`.

**`backend/harness/tools/orchestrator_tool.py`** —
`_create_kanban_board` now accepts `session_id` and stashes it in
`config.session_id`. `cmd_orchestrate` and the `OrchestrateTool` pass
it through. The board's `config.session_id` is the routing key for
the EventSourceSink so `board.completed` events reach the right
session's subscribers (vs broadcasting to everyone).

**`backend/harness/orchestrator.py`** — `_wait_for_board` is now a
thin shim over `BoardWaiter.wait()`. Preserves the legacy return
shape (`success`, `board_id`, `tasks`, `stalled_tasks`,
`blocked_tasks`, `elapsed_seconds`) and adds `method` and
`events_received` for observability. Passes `session_id` to
`cmd_orchestrate` so the board is wired to the orchestrator's SSE
queue from creation.

**`backend/harness/api/state.py`** — added `set_event_source_sink()`
and `get_event_source_sink()` so the orchestrator (no `Request`
object) can subscribe to the same per-session queue the SSE router
uses.

**`backend/api/main.py`** — calls `set_event_source_sink(sink)` at
startup right after `set_event_bus(bus)`.

## Behavior change (observable)

For a 30-minute orchestration with 100 tasks that complete normally:

| | Before C03 | After C03 |
|---|---|---|
| DB polls | ~120 (one every 15s) | 0 (push) or 1-2 (silence poll, never fires) |
| Latency from last task to "board done" | up to 15s | <100ms (push) |
| Per-task progress visible to orchestrator | at next poll | immediately via `subagent.completed` event |

The 5400s cap is preserved (so a genuinely stuck run still times out
and the orchestrator can fall back to manual intervention).

## Failure modes

| Failure | Detection | Behavior |
|---|---|---|
| `cmd_orchestrate_monitor` raises | caught in `_poll_once`, log debug | `_poll_once` returns None, loop continues |
| EventSourceSink torn down (shutdown) | `unsubscribe` raises | caught in `wait()` finally block, logged |
| Subagent emits `subagent.completed` repeatedly | deduped by `(id, status)` set | logged once per pair |
| Event for wrong board/session | filtered in `_classify_event` | counted in `events_received` but doesn't trigger return |
| No events at all (subagent crashed silently) | 60s silence poll catches it | DB poll discovers terminal state or returns in_progress |
| `board.completed` event but tasks list is empty | OK | push still works, dashboard just shows "all done" with no detail |
| `_emit_board_completion_if_done` raises | caught, log warn, swallow | Worst case: missed event, silence poll catches within 60s |

## Open questions

1. **Should `subagent.completed` errors terminate the wait?** Currently
   we just log a warning and wait for the kanban service to escalate
   to `board.failed`. Alternative: surface per-subagent errors as
   "degraded" status. Decision: defer — current behavior is safer
   (a single failed subagent shouldn't tear down the whole board).

2. **Should the push path disable the silence poll entirely when the
   sink is healthy?** No — the sink is per-session but the
   `subagent.completed` events come from child sessions, which route
   back to the parent's session_id. The poll is a safety net for
   edge cases (e.g. EventBus drained too aggressively). Keep it.

3. **Multi-board waits**: if a single run creates multiple boards
   (e.g. an explore + build split), the orchestrator would need to
   wait on both. Not currently a need — the orchestrator creates
   one board per run. If we later split, the BoardWaiter can be
   fanned out and results merged in the orchestrator.

4. **What if the kanban DB is unavailable when the silence poll fires?**
   `cmd_orchestrate_monitor` returns `{"error": "Database not connected"}`
   on no-DB. `_poll_once` treats that as "not terminal" and returns
   None. The loop continues. Eventually the cap (5400s) fires. This
   matches the pre-C03 behavior — no regression.
