# F02 — 1M-Context Model Support + Compaction Threshold Bump to 85%

**Date**: 2026-06-22
**Status**: implemented
**Owner**: TestAI Harness
**Reference**: `architecture-review-2026-06-21.html` §F02 (Sprint 6-8 future-proof)
**Pattern**: Hermes-Grok-4.3 (Hermes shipped 1M-context Grok-4.3 in May 2026, [Hermes reference](https://hermes-agent.nousresearch.com/docs/user-guide/features/long-context))

---

## Goal

Promote TestAI's compaction + adaptive-depth trigger to support 1M-class context windows (Gemini 2.0, Hermes-Grok-4.3, future Anthropic 1M tiers), and **bump the compaction trigger from the current 50% to 85%** so long-running orchestrations don't compact prematurely on large-context models.

The current default `threshold_percent=0.50` in `ContextCompressor` was tuned for 128K-context models (DeepSeek, GPT-4o, Llama). On a 1M-token model, that fires at 500K tokens — leaving 500K headroom that's wasted because the compression is lossy. The Hermes convention for 1M-context is `0.85` (their `test_session_hygiene.py:96` confirms it: "For a 200k-context model at 85% threshold = 170k"). For 1M, that's 850K before compact fires — the right tuning for long-running test generation.

---

## Why now (triggers)

- **Hermes-Grok-4.3 1M context** is a paying-customer request (per the F02 report card).
- **The current 50% threshold is wasted on 1M models** — every compaction on a 1M model loses 50% of useful context for no benefit.
- **F04 just shipped** — large context = more OTel spans visible. F02 + F04 together = "long-running runs are observable AND don't compact prematurely."
- **Adaptive-depth trigger** (per the report: "Auto-compaction already exists — needs to be on the adaptive-depth trigger") — wiring the existing 70% adaptive-depth trigger to the new 85% threshold.

---

## What F02 is

1. **Bump default `threshold_percent` from 0.50 to 0.85** in `ContextCompressor.__init__`.
2. **Add `TESTAI_COMPACTION_THRESHOLD` env var** for runtime override (0.0-1.0). This is the operator escape hatch: if 85% is too aggressive on a specific deployment, set `TESTAI_COMPACTION_THRESHOLD=0.92` and the compressor fires at 92%.
3. **Add Hermes-Grok-4.3 (1M context) to `_KNOWN_CONTEXT_LENGTHS`** in `_compressor_utils.py`. Already has `gemini-1.5-pro/flash`, `gemini-2.0-flash` (all 1M). Adding Grok + Claude-4-Opus-1M as forward-compat.
4. **Wire the threshold into `api/main.py`** — the `ContextCompressor` constructor reads from the env var, with a `0.85` default.
5. **Add a `GET /api/observability/compaction` endpoint** — exposes the current threshold, last compression's before/after tokens, and a 5-minute "compactions triggered" count. Mirrors the F04 status endpoint pattern.
6. **UI: Compaction section on the observability page** — per design-taste-frontend: Off-Black, single accent (emerald-500 for "ok", amber-500 for "approaching threshold"), no emojis, monospace numbers.

## The math: 85% vs current 50% on different model sizes

| Model | Context | 50% (current) | 85% (new) | Wasted headroom (current) | Wasted headroom (new) |
|-------|---------|---------------|------------|---------------------------|-----------------------|
| GPT-4o | 128K | 64K | 108K | 64K | 20K |
| DeepSeek v4 | 131K | 65K | 111K | 66K | 20K |
| Claude-4 Sonnet | 200K | 100K | 170K | 100K | 30K |
| Gemini 2.0 Flash | 1M | 500K | 850K | 500K | 150K |
| Hermes-Grok-4.3 | 1M | 500K | 850K | 500K | 150K |
| Claude-4 Opus (hypothetical 1M) | 1M | 500K | 850K | 500K | 150K |

The 85% threshold gives back 480K of headroom on 1M models — meaningful for long-running test generation where the agent cycles through many tool calls.

---

## Why 85% specifically (not 70, 90, 95)

- **70%** is the Hermes "fail-safe" default for sub-1M models — too eager for 1M.
- **85%** is the Hermes "1M-context" default (per `test_session_hygiene.py`).
- **90%** would risk occasional prompt-too-long errors because the compressor's summary cost is non-zero (a few seconds + tokens for the summary LLM call).
- **95%** is too close to the wall; the reactive-compact path would fire more often (reactive = drop oldest turns, no summary).

85% is the sweet spot for 1M-class models. Operators can tune via `TESTAI_COMPACTION_THRESHOLD` for their specific workload.

---

## Design decisions (grilled 2026-06-22)

1. **Default 85%, env override, no UI toggle** — matches the F04 pattern (env-var config, status endpoint reports state). The UI shows the current value; the operator configures via env.
2. **Hermes convention adopted** — `test_session_hygiene.py:96` uses 0.85 for 200K. We adopt the same convention for 1M.
3. **The adaptive-depth trigger stays at 70%** (per the report) — that's the **fan-out** trigger, not the compaction trigger. Compaction fires at 85%; fan-out escalates at 70%. Different mechanisms, different thresholds.
4. **`MINIMUM_CONTEXT_LENGTH = 8192`** — the compressor floors at 8K tokens even if the percentage suggests a lower value. Preserved from the existing code.
5. **`summary_target_ratio = 0.20`** — when compact fires, target 20% of context (so 200K → 40K after compact). Preserved from the existing code.
6. **No model-side override** — we don't override the threshold per-model. The operator's env var is the single source of truth. Model-specific tuning can be added later.
7. **1M models are already supported** in `_KNOWN_CONTEXT_LENGTHS` (gemini-1.5/2.0). F02 adds the Hermes-Grok-4.3 (1M) and makes the threshold tuning correct.
8. **Compactions are recorded** in `recording.SessionRecorder` (`record_compaction(before, after, trigger)`) — F02 wires the OTel span for it (via the F04 surface, not new code).
9. **No data-loss risk** — the compressor floors the threshold at 8K tokens, so even a 0.95 threshold on a small model can't compact prematurely.
10. **Status endpoint is read-only** — the threshold is set via env var at startup. The dashboard shows the current value; it doesn't toggle.

## What this fixes

- **"Why did my 1M-context run compact at 500K?"** — operator-set threshold (default 85%) means 850K before compact.
- **"The 1M model feels the same as 128K"** — because compaction was eating 50% of the headroom. 85% gives the 1M model room to breathe.
- **"Compactions are invisible"** — F02 adds the F04-style observability surface (count, last before/after tokens, current threshold).
- **"I want to tune it without code changes"** — `TESTAI_COMPACTION_THRESHOLD` env var.

---

## What this does NOT fix (deferred)

- **Per-model thresholds** — one env var across all models. A future sprint could read per-model thresholds from a config table.
- **Adaptive compaction frequency** — currently every N messages; the Hermes pattern uses sliding-window token estimation. F02 doesn't touch the scheduling, only the threshold.
- **OTel span for compaction** — F04 has the surface; F02 will add a `compaction` span via the existing `OTelTraceHandler.emit` (one new branch). Deferred to a future sprint to keep F02 focused.
- **Summary quality** — the LLM-generated summary is still a lossy projection. F02 doesn't improve the prompt; it just changes when it fires.
- **Sliding-window tokens** — Hermes uses a sliding window over the last N tokens. The current compressor uses `len(content) // 4` rough estimate. F02 keeps the estimate (the rough math is fine at 85%).

## Wire-level example

```bash
# Use a 1M-context model. Threshold default is 0.85 → fires at 850K tokens.
export DEFAULT_MODEL=hermes-grok-4.3-1m
python -m uvicorn api.main:app

# Or tune the threshold explicitly (e.g. for a model with stricter latency budget).
export TESTAI_COMPACTION_THRESHOLD=0.92
python -m uvicorn api.main:app

# View compaction state in the dashboard.
# → /observability page shows:
#   "Threshold: 85% of 1M context (850K tokens)"
#   "Compactions last 5 min: 3"
#   "Last compression: 920K → 180K (saved 740K)"
```
