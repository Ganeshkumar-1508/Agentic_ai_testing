# C06 — Subagent heartbeat + stale detection

**Date**: 2026-06-21
**Status**: Implemented
**Decision tree**: `docs/2026-06-21-architecture-decision-tree.md#c06`
**Reference**: `reference/hermes-agent/tools/delegate_tool.py:1318-1684`

## Problem

`delegate_task` blocks the parent on a subagent. If the subagent hangs
on a long-running tool (`apt-get`, `web_fetch`, network wait), the
parent's gateway inactivity timeout can fire — killing a healthy
subagent. The same risk exists for any parent that talks through a
gateway with its own timeout.

Additionally, there was no way to detect a stuck subagent. A child
that loops on a misbehaving tool, hangs in a subprocess, or gets
swapped out by the OS would silently consume budget until the
overall run cap (5400s) fired.

## Decision (locked, 4 questions)

| Q | Decision | Rationale |
|---|----------|-----------|
| Q1 | Heartbeat location: inside `delegate_task._run_single_enhanced` | The heartbeat is part of the delegate contract; the orchestrator doesn't need to know about it. TestAI's asyncio loop means we can use `asyncio.create_task` instead of Hermes' `threading.Thread`. |
| Q2 | Hermes defaults: 5s interval, 6 idle cycles, 60 in-tool cycles | 5s interval = 30s idle fuse, 5min in-tool fuse. Both are configurable via env vars for tests / tight deployments. |
| Q3 | Raise `SubagentStuckError` (parent decides) | Heartbeat is for observability, not termination. The parent has context to decide: re-plan, escalate to user, or accept a slow run. The existing `delegate_task` error path turns the error into a structured `SubagentResult` with `status="error"`. |
| Q4 | Progress signal: `stream_events`-style = `(api_call_count, current_tool)` pair from `Agent.get_activity_summary()` | TestAI doesn't have Hermes' `get_activity_summary`; we add the same shape. Tool change (same iter, different tool) also counts as progress — the child is alive even mid-round. |

## Implementation

### Modified: `backend/harness/agent/agent.py`

Added to `Agent.__init__`:
- `_api_call_count: int` — incremented at the start of each LLM call
- `_current_tool: str | None` — set before tool execution, cleared after
- `_last_activity_desc: str` — human-readable description of the current activity

Added 3 new methods:
- `get_activity_summary() -> dict` — returns `{current_tool, api_call_count, max_iterations, last_activity_desc}`. Falls back to deriving `current_tool` from `_messages` if the explicit counter is `None` (e.g. resumed session).
- `_set_current_tool(name)` — internal helper for `run_stream` to stamp "in tool" / "between tools" state.
- `_bump_api_call()` — internal helper for `run_stream` to count LLM calls.

Modified `run_stream` to:
- Bump `_api_call_count` at the start of each round (line 552+).
- Set `_current_tool` before tool execution (both sequential and concurrent paths).
- Clear `_current_tool` after tool execution completes (line 762+).

### New file: `backend/harness/services/heartbeat.py` (~270 lines)

The `SubagentHeartbeat` class:

```python
class SubagentHeartbeat:
    def __init__(self, subagent_id, target, *, interval=5.0,
                 stale_cycles_idle=6, stale_cycles_in_tool=60,
                 on_heartbeat=None, on_stale_warning=None): ...

    async def run(self, stop_event: asyncio.Event) -> HeartbeatOutcome:
        while not stop_event.is_set():
            await asyncio.wait_for(stop_event.wait(), timeout=self._interval)
            # poll target.get_activity_summary()
            # track (iter, tool) pair
            # if unchanged for stale_limit cycles → return HeartbeatOutcome(stuck=True)
            # else → fire on_heartbeat callback
            # half-threshold → fire on_stale_warning (once)
```

The `SubagentStuckError` exception carries the failure payload
(`subagent_id`, `last_iter`, `last_tool`, `stale_seconds`) so the
parent can build a structured `SubagentResult` for the LLM.

### Modified: `backend/harness/tools/delegate_task.py`

Inside `_run_single_enhanced`:
1. After creating `child_task`, instantiate `SubagentHeartbeat` with the child as target.
2. Spawn `heartbeat_task` via `asyncio.create_task`.
3. Race `child_task` vs `heartbeat_task` vs `cancel_task` in the existing `asyncio.wait` set.
4. If `heartbeat_task` finishes first with `stuck=True`:
   - Cancel `child_task`.
   - Persist the failure to `agent_delegations` with `status="error"`.
   - Record the failure in the provider-quality tracker.
   - Return a `SubagentResult(status="error", error="SubagentStuckError: ...", detector_name="heartbeat_stale")`.
5. On clean child return: set `heartbeat_stop`, await `heartbeat_task` to drain.
6. The `finally` block also stops the heartbeat (defensive — covers the cancel/exception paths).

The per-cycle `on_heartbeat` callback emits a `subagent.heartbeat`
event on the `EventBus` so the dashboard's SSE feed shows a
live "heartbeat" stream for each active subagent.

## Behavior change (observable)

| Scenario | Before C06 | After C06 |
|---|---|---|
| Subagent running `apt-get` for 4 minutes | Parent killed by gateway timeout | Parent sees continuous heartbeat events; no false kill |
| Subagent hangs (no progress) | Detected at 5400s (the cap) | Detected at 30s idle / 5min in-tool |
| Subagent failed (e.g. OOM mid-tool) | Detected at next poll cycle of `_wait_for_board` (up to 60s with C03) | Detected within 5min; `SubagentResult.status="error"` |
| Dashboard observability | No per-subagent live signal | `subagent.heartbeat` event every 5s on SSE |

## Failure modes

| Failure | Detection | Behavior |
|---|---|---|
| Child's `get_activity_summary` raises | caught in heartbeat, log debug | Treated as "no progress" → stale counter bumps |
| Heartbeat callback raises | caught in heartbeat, log debug | Heartbeat continues; callback dropped |
| Stop event set during the wait | `asyncio.wait_for` returns | Loop exits cleanly with `stuck=False` |
| Both child and heartbeat finish simultaneously | `asyncio.wait` returns both | We pick child (success), cancel heartbeat |
| Stale-warning callback raises | caught, log debug | Warning dropped, but heartbeat continues |
| Heartbeat init fails (import error etc.) | caught at delegation time | Run continues without heartbeat (best-effort) |

## Configuration

| Env var | Default | Description |
|---|---|---|
| `SUBAGENT_HEARTBEAT_INTERVAL` | `5.0` | Seconds between heartbeat cycles |
| `SUBAGENT_HEARTBEAT_STALE_IDLE` | `6` | Stale cycles before raising when idle |
| `SUBAGENT_HEARTBEAT_STALE_IN_TOOL` | `60` | Stale cycles before raising when in a tool call |

All three are also direct constructor params on `SubagentHeartbeat`.

## Event payload

`subagent.heartbeat` event (emitted on the EventBus every cycle):
```json
{
  "type": "subagent.heartbeat",
  "session_id": "...",
  "data": {
    "subagent_id": "sa-...",
    "api_call_count": 7,
    "current_tool": "web_fetch",
    "last_activity_desc": "running tool web_fetch",
    "stale_count": 0,
    "elapsed_seconds": 35.2
  }
}
```

The dashboard can show a live "X subagents active, Y idle, Z in tool"
widget by aggregating these events per session.

## Open questions

1. **Should the heartbeat also signal cancellation grace?** Today, the
   heartbeat polls at 5s; a long tool could finish mid-poll. If the
   parent wants to cancel a child, it currently has to wait up to
   5s for the next poll to notice. A 1s "wakeup on cancel" variant
   would be cleaner. Defer — current behavior matches Hermes.

2. **Multiple concurrent subagents**: the heartbeat is per-subagent
   in `delegate_task`, so N concurrent subagents = N heartbeat tasks.
   The cost is 5s × N per parent, negligible. No batching needed.

3. **Sub-stuck detection (sub-tool):** if a single tool call is hung
   (e.g. `bash` is waiting on stdin), the heartbeat sees
   `current_tool="bash"` and uses the in-tool threshold (5min). This
   matches Hermes' design intent. A future C18 deepening could add
   a per-tool wall-clock budget.

4. **Integration with the budget tracker:** the heartbeat could feed
   the per-subagent budget tracker the wall-clock time of "active
   thinking" vs "in tool" — useful for cost analysis. Defer.

5. **Heartbeat is in-process only:** if the orchestrator crashes
   mid-subagent, the child is orphaned. The existing 5min
   `reclaim_sessions` job in `orchestrator.py` handles this at the
   container level, not the heartbeat level. C01 (worktree) will
   add per-subagent isolation that makes orphans cheaper.
